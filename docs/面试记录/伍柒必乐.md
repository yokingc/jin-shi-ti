React Router 主要用于 SPA 的页面切换和路由管理。实际项目里我一般会用它做页面路径映射、动态路由参数获取、嵌套路由拆分，以及配合懒加载做按路由维度的资源加载。像列表页跳详情页，我会通过 useNavigate 跳转，通过 useParams 或 search 参数拿业务 ID。

redux：
单向数据流：组件 dispatch action → reducer 处理 → store 状态更新 → 组件重新渲染
核心思想是：
全局状态放在一个 store 里
不能直接改状态
必须通过 action 描述“要做什么”
reducer 根据 action 返回新状态

react-reduex：
1）useSelector从 store 里取数据
2）useDispatch派发 action
3）Provider把 store 注入应用

高频追问：
1.React Router 底层原理你知道多少：1.监听路由变化。2.维护当前 location 状态。3.做路由匹配并渲染对应组件。
BrowserRouter 依赖的是 H5 History API，HashRouter 依赖的是 URL hash 变化，也就是 # 后面的内容变化不会真正请求服务端。
2.Redux 单向数据流指的是：view 触发 action → dispatch 到 store → reducer 计算新 state → view 基于新 state 重新渲染。让状态变化可预测、可追踪、可调试。
3.Redux 和 Context 区别：都能做跨层级数据传输，但定位不同。conText适合比较稳定、层级较深、逻辑不复杂的问题。redux是更完整的状态管理方案，除了共享数据还提供统一store，更新粒度和规范，中间件，devtools处理。
4.useMemo和useCallback区别：useMemo缓存函数结果，useCallback缓存函数本身，实践中经常和react.memo使用。不能滥用。
5.Webpack loader 和 plugin 区别：loader 本质上是 文件转换器，webpack本身只认js和json。plugin 是 插件机制，介入webpack的整个打包生命周期，提供功能扩展。
6.Babel 做了什么转换：Babel 本质上是 JavaScript 编译器。Babel 负责语法降级如ES6、jsx、TS等，polyfill 负责补运行时缺失能力，比如一些数组能力。



人物角色：你承担的角色和普通开发人员有什么区别，与TL的区别
思考：SA（System Architect） 更偏向负责一个业务域内的技术方向和架构治理。相比普通开发，SA 不只是完成具体功能开发，还需要对负责域内项目的技术选型、长期演进、架构稳定性以及防腐化能力负责。同时，SA 也会更早介入需求阶段，参与需求预审、方案把关、排期评估以及人力协调，确保项目从设计到落地整体可控。TL 则更偏团队管理和横向协同。除了关注技术落地外，TL 还需要承担更多跨团队、跨领域的协调工作，并对人员管理、团队建设、任务分配和整体交付结果负责。

AI相关：
了解最近AI前沿的东西吗 -> harness harness不是单指某个模型或某个产品，它更像 AI agent 的“运行外壳 / 控制层”，也可以说是agent工程化。Anthropic 把它描述成让模型真正以 agent 方式运行的 wrapper，负责提示词组织、工具调用、上下文续接、循环执行这些事情.和前端结合的话可以这样理解，将prd+设计规范+组件库+代码仓库|playwright、CI检查等接近同一个harness，模型不再是生成代码而是按照流程化从读需求跑用例修bug一条龙闭环。 MCP模型上下文协议，扩展了agent的能力，比如前端取获取prd、接口文档、代码仓库等。computer use/browser agents，可以让模型直接操作浏览器界面，适用于页面联调、复杂交互回归、bug复现和E2E自测。

对前端来说，E2E 是测试范围，强调从用户视角把完整链路跑通，Playwright 更适合把核心流程固化成稳定脚本，而 browser agent / computer use 更适合探索式测试、复杂交互自测和 bug 复现。长期看，我觉得更有价值的是两者结合，而不是互相替代。

了解AI的一些概念吗，比如大模型参数：
1.参数（Parameters） 本质上就是模型里的大量数字权重，训练的过程就是调整这些权重并让模型对下一个token预测越来越准。参数多不一定代表模型更聪明，但训练和推理和成本越高。
2.token，是大模型的实际处理文本，具体取决于分词方式，由token延伸出上下文长度等概念。
3.上下文窗口（Context Window） 就是模型一次能“看到”多少输入内容。

训练、微调、推理：训练是学通识能力，微调是做领域适配，推理是实际使用阶段。对业务落地来说，大多数团队更多是在推理层和应用层做优化，而不是自己从头训练大模型。

完整回答：了解一些。比如大模型里的参数，我理解就是模型在训练过程中学到的权重，本质上决定了模型怎么把输入映射成输出。参数量越大，通常说明模型容量更强，但不代表效果一定更好，因为还和数据质量、训练方法、上下文能力、推理成本有关。
除了参数，我也会关注几个基础概念，比如 token，也就是模型处理文本的基本单位；上下文窗口，也就是模型一次能看到多少内容；还有训练、微调和推理这几个阶段。
我现在更关注的是这些概念怎么和实际落地结合，比如在前端场景里，选模型不只是看参数大不大，还要看生成代码是否稳定、上下文够不够、成本和响应速度能不能接受。
Embedding：把文本转成向量，方便检索和召回 -> 向量库，RAG的基础
RAG：先检索外部知识，再让模型回答，减少幻觉
Hallucination：模型会一本正经地说错话
Tool Calling：模型调用外部工具，不只靠自己“猜”
Fine-tuning微调 vs Prompting：一个是改模型，一个是改输入方式.很多团队实际落地时，往往先用 prompting + RAG + tool calling，只有当这些手段还不够时，才考虑 fine-tuning。

微前端：
我理解微前端的核心目标是一致的，都是把前端系统拆成可独立演进的业务单元。只是社区里更常见的是 single-spa、qiankun 这种主应用编排模式，主应用按路由去注册、挂载、卸载子应用。
我司这套更偏 App 容器驱动的微前端。Native 会先预热一个 microweb 容器，页面跳转时把 URL 和部分配置传给容器；容器再解析路由，读取对应项目的 micro.json，动态加载依赖和页面 JS，最后从 window 上拿到 UMD 模块挂载到 Vue/React 容器里。
所以它本质上仍然是微前端，只是更偏运行时装载和平台化治理，而不是纯 Web 场景下的主应用注册子应用。它还额外集成了 Bridge、缓存、灰度、预渲染、prefetch、预请求，甚至离线能力，这也是它和网上常见方案差异比较大的地方。
