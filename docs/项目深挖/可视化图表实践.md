# 可视化图表实践（工作台 + 询价模块）

## 一、做了什么

### 工作台 `business-overview.vue`

- 用 ECharts 渲染两个**面积折线图**：企业收支趋势 + 项目计划趋势
- `LinearGradient` 做渐变填充，tooltip 自定义 HTML 格式（标题 + 多系列数据）
- 实现了 `dataZoom`：数据超过 10 条时自动出现底部滑动条，固定窗口为 10 条，支持鼠标滚轮缩放
- `beforeDestroy` 调用 `dispose()` 释放实例，防止内存泄漏

### 询价模块 `price-chart.vue` + `supply-demand-chart.vue`

- **价格趋势图**：柱状图 + 折线图双 Y 轴，支持分位切换（50/75分位等），切换时 `setOption(option, true)` 全量刷新避免数据残留
- **供需比图**：双折线（实线 + 虚线），自定义 tooltip 展示供需比和常规市场基准值
- 两个组件都用了 `ResizeObserver` 监听容器尺寸变化 + `window.resize` 双重保险
- 数据从无到有时用 `$nextTick` 延迟初始化，避免 DOM 未挂载就 `echarts.init`

## 三、优化方向 + 触发场景

### 1. Resize 防抖

**现状问题**：resize 回调没有防抖，窗口拖动时高频触发 `chart.resize()`，每次都触发 ECharts 内部重绘。

**触发场景**：用户拖动浏览器窗口、侧边栏收起展开、响应式布局切换。

```js
// 优化方式：用 lodash debounce 包一层
handleResize: _.debounce(function () {
  if (this.chart) this.chart.resize()
}, 200)
```

**追问延伸**：为什么用 debounce 不用 throttle？
> resize 场景关心的是"停止拖动后的最终尺寸"，不需要中间过程，debounce 更合适。throttle 适合需要中间帧的场景，比如滚动加载。

---

### 2. ECharts 按需引入

**现状问题**：`import * as echarts from 'echarts'` 全量引入，打包体积约 1MB+。

**触发场景**：项目图表类型固定（只用折线图、柱状图），没必要打包地图、3D、关系图等模块。

```js
// 优化方式：按需引入
import * as echarts from 'echarts/core'
import { LineChart, BarChart } from 'echarts/charts'
import { GridComponent, TooltipComponent, LegendComponent, DataZoomComponent } from 'echarts/components'
import { CanvasRenderer } from 'echarts/renderers'

echarts.use([LineChart, BarChart, GridComponent, TooltipComponent, LegendComponent, DataZoomComponent, CanvasRenderer])
```

**效果**：打包体积可减少 60-70%，首屏加载更快。

**追问延伸**：什么时候用 SVG renderer 替代 Canvas？
> 数据量小、需要导出高清图片、或者需要 CSS 动画时用 SVG。数据量大（>1000 个点）、需要高性能渲染时用 Canvas（默认）。

---

### 3. 大数据量 sampling（降采样）

**现状问题**：数据量小时没问题，但如果数据增长到几千条，折线图每个点都渲染会导致掉帧。

**触发场景**：时间跨度拉长（从月视图改为年视图）、实时数据持续追加、后端返回明细级数据。

```js
// 优化方式：ECharts 内置 LTTB 算法
series: [{
  type: 'line',
  sampling: 'lttb',  // Largest-Triangle-Three-Buckets，保留视觉特征的降采样
  data: largeDataArray
}]
```

**追问延伸**：LTTB 是什么原理？
> 把数据分成等量的桶，每个桶里选一个点，选择标准是让相邻三个桶的三角形面积最大，这样能最大程度保留折线的视觉形状，比简单的均匀采样效果好很多。

**更激进的方案**：超过 10 万条数据时，可以考虑 WebGL 渲染（ECharts GL）或者 Canvas 直接绘制，绕过 ECharts 的 DOM 管理开销。

---

### 4. 数据缓存（询价场景）

**现状问题**：用户反复查同一条线路（比如上海→北京，4.2米厢式），每次都重新请求接口。

**触发场景**：询价是高频重复操作，同一用户在一次会话里可能查同一条线路多次。

```js
// 优化方式：内存 Map 缓存，带 TTL
const cache = new Map()

async function fetchWithCache(key, fetcher, ttl = 5 * 60 * 1000) {
  const cached = cache.get(key)
  if (cached && Date.now() - cached.timestamp < ttl) {
    return cached.data
  }
  const data = await fetcher()
  cache.set(key, { data, timestamp: Date.now() })
  return data
}

// 调用
const key = JSON.stringify(queryParams)
const result = await fetchWithCache(key, () => getSingleInquiry(queryParams))
```

**追问延伸**：为什么不用 Vuex 缓存？
> Vuex 适合全局共享状态，询价缓存是局部的、临时的，用 Map 更轻量，也不会污染全局 store。如果需要跨页面共享或持久化，才考虑 Vuex + localStorage。

---

### 5. WebSocket 实时推送（如果有实时需求）

**当前没有**，但如果工作台需要实时刷新（比如订单数实时变化），可以这样设计：

```js
// 思路：WebSocket 推送增量数据，前端只更新 diff
mounted() {
  this.ws = new WebSocket('wss://api.example.com/workbench/realtime')
  this.ws.onmessage = (event) => {
    const delta = JSON.parse(event.data)
    this.applyDelta(delta)  // 只更新变化的字段，不重新渲染整个图表
  }
},
beforeDestroy() {
  this.ws?.close()
}
```

**追问延伸**：WebSocket 断线重连怎么做？
> 监听 `onclose` 事件，用指数退避策略重连（1s → 2s → 4s → 最大30s），避免服务端压力。同时维护一个心跳 ping，检测连接是否存活。

**追问延伸**：WebSocket 和 SSE（Server-Sent Events）怎么选？
> 只需要服务端推送（单向）用 SSE，更简单，自动重连，走 HTTP 不需要额外协议。双向通信（客户端也要发消息）才用 WebSocket。工作台数据推送用 SSE 就够了。

>
> 这个项目数据量不大，没有遇到大数据渲染的性能瓶颈，也没有 WebSocket 实时推送的场景。如果要优化，我会考虑：resize 回调加防抖、ECharts 按需引入减少包体积、大数据量时开启 sampling: 'lttb' 降采样、以及对高频查询结果做内存缓存。"
