1.讲一下你如何利用AI进行code review / 双门禁机制？

非孤立->体系->AI coding暴露的问题->双层门禁->第一层husky githooks+ 第二层 gitlab push hook+AI review->分别解决什么问题
开头：讨论AI review时，我一般不会把它当成一个孤立工具，我会把它放到整个质量体系里讲。因为我们当时推进 AI 开发后，最大的风险不是‘能不能生成代码’，而是‘生成得快以后，质量怎么兜住’。所以我们做的不是单点 AI 审查，而是一套双层质量门禁：第一层是本地 Husky + Git Hooks，先拦住类型、Lint、提交规范这类确定性问题；第二层是合并前的 AI Review + GitLab Push Hook，去补足复杂度、重复逻辑、潜在 Bug、性能隐患这类传统规则不容易覆盖的问题。整个体系跑下来，单月执行了 56 次代码审查，识别了 207 个问题。

追问：具体review什么内容？review有边界/限制吗？
我会把AI review定义为合并前的语义预审层，并不是完全代替人工cr，我把他放到GitLab Push Hook 之后，和 ESLint 增量检查并行跑，先把高风险问题前置暴露出来，再由评审人做人工复核闭环。
具体review的，一般分为两类，第一类是规则性、确定性的问题，主要是第一道门禁来拦截，第二道做兜底，比如类型、lint、提交规范等；第二类是语义型、工程型问题，这一部分是AI review的重点，比如复杂度、重复逻辑、潜在bug、性能隐患等，以及风格和既有实现是否一致。
但它也有边界。第一，它不替代人工，试点里我们就是按 AI 预审、人工复核闭环；第二，它不负责替产品判断需求是否正确，这类问题应该前置在 PRD 澄清和方案阶段解决；第三，它不可能覆盖所有运行时和联调问题，像接口变更、时序问题、脏数据问题还是得靠联调、自测和线上监控。”
总的来说，AI review不是代替人工cr，而是解决面对大量diff代码的注意力成本问题，通过语义预审帮助评审人关注高风险问题，降低漏看率，通过这种AI预审+人工闭环的模式我们单月56次CR重识别出200多个潜在问题。

追问：重点讲一下review的流程，评审反馈如何反哺给review系统？
AI Review 是远端合并前门禁。MR 或 push 到 release 后，GitLab Webhook 进服务端，先做项目和 MR 校验，再丢 BullMQ 队列。消费时会读取项目规则、分配评审人、拉取 MR diff 和文件内容，然后并行跑 AI 语义审查和 ESLint 增量检查。AI 审查内部是两阶段：先用模型根据文件、diff 和规则列表找疑似问题行，再按 ruleId 进入具体规则节点，用 detectionPrompt 精查，最后输出行号、问题描述、修改建议和严重等级。结果写库后发钉钉，评审人到 dashboard 复核。评审反馈这块，如果标记为误报，会触发 false-positive 流程，把规则名、代码片段、错误信息推到内部群；目前不是自动训练模型，而是人工根据误报样本调整数据库里的规则 prompt、severity 或启停状态，下一轮审查读取新规则，这就是反馈反哺 review 系统的方式。

追问：如果AI review误报、漏报怎么处理？
首先AI一定是有误报的和漏报的，我们并没有完全信任他所以才会有AI预审+人工复核的形式。
误报的思路：1.不然AI直接阻断合并，而是先进钉钉的dashboard给评审人复核 2.确定性问题尽量交给本地门禁去拦截，避免AI花太多注意力关注这类高误报场景
漏报的思路：1.多层兜底，本地+AI+人工 
另外，dashboard设计了反馈机制，评审人可以对问题进行反馈，收集后可以将这些规则沉淀下来反哺给AI review系统，持续提高AI预审的准确率。

追问：数据库在哪、怎么通信？服务端和前端的职责是什么？
数据库类型MySQL，连接串通过环境变量注入。
通信方式：NestJS 服务端用 Prisma Client 直连，所有的表操作都是 Prisma生成的。
服务端这边负责接 GitLab Webhook、做前置校验、入 BullMQ 队列、拉 MR diff 和文件内容、加载规则、并行执行 AI 语义审查和 ESLint 增量检查、写库、发钉钉通知；前端这边主要是 toolkit dashboard，负责把服务端产出的 review 结果展示给评审人，并承接误报、已修复、wont-fix 这类人工判定操作，推动流程闭环。
服务端负责算：拉代码、跑规则、调模型、产出问题。
前端负责看和改状态：展示问题列表、严重级别、建议，供评审人确认。
最终闭环还是在服务端：前端的操作会回传给服务端更新状态，所有问题处理完后，服务端再把流程置为完成，并发送完结消息。


讲一下本地门禁的流程：
本地门禁的定位不是替代AI review，而是先把规范类型格式等高确定性问题拦截在本地，减少远端审查压力。
本地门禁本质上就是 Husky + Git Hooks 挂在开发者本地的 commit / push 链路上，目标是把确定性问题尽量前置拦住。具体来说，pre-commit 和 pre-push 都会先跑 pnpm -w run type-check 做全局 TypeScript 类型检查；同时 lint-staged 只对本次暂存改动做增量处理，像 js/ts 文件会执行 eslint --fix + prettier --write，json/md/css/html 这类文件主要走 Prettier；另外还有 commitlint 校验提交信息格式，要求是 type(scope): subject。只要任一环节失败，就直接阻断提交或推送，开发者得先在本地修好再继续。
ESLint 这块不是每个项目自己零散配一堆规则，而是通过 .eslintrc 去继承统一配置链，里面除了 React、TypeScript 推荐规则，还有一些团队自定义规则，比如限制函数长度、if 嵌套深度、变量命名规范、JSON.parse 必须包 try-catch 等，所以本地门禁既有通用规范，也有业务侧工程约束
