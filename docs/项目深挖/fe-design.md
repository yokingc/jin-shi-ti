3.展开讲一讲fe-design（最具代表性的工作）和ai review（质量门禁、code review），这些是你做的吗？

fe-design：我最有代表性的工作是自研了 fe-power-prd CLI。因为我们试点后发现，真正影响后续效率的往往不是编码本身，而是前面的 PRD 理解和方案初稿质量。我做这个工具，就是想把这段前置工作标准化：输入 PRD，自动生成前端设计文档和疑惑点清单，再通过自检、澄清、回灌形成闭环。这样团队里不同人面对同一份 PRD，输出就不会差异那么大。后面我又补了评估样本、跨平台适配和结构化 DSL，让它从一个个人工具，变成团队能复用的基础能力。

追问1：为什么需要做自检、澄清、回灌的机制，有什么效果？14组真实样本和评估结果有什么作用？
核心问题原因是因为在试点的过程中，我发现阻塞点不是编码阶段，而是prd理解阶段的失真的层层传导。如果前面功能点拆错了、边界不明晰、场景遗漏等，到了后面都会存在不稳定，容易产生返工和人工介入，耗费时间和心智。所以设计fe-design时，设计的思路不是快而是稳，通过自检、澄清、回灌机制把流程的头结点产物输出得更稳定、更可控。自检主要是为了解决功能点拆的过粗（多页面多操作被视为一个功能点）、漏掉边界两个问题（只关注了主流程，没关注支线操作）。澄清主要是解决真实prd天然有很多模糊点，职责分配不清，避免ai直接接手的幻觉问题，澄清会把置信度低的点、待确认规则、边界缺失收集起来，用户可当面交互澄清也可以选择生成问题文档手动澄清后继续。回灌主要是把澄清的点一次性维护到设计文档中，因为后续步骤都是依赖这个文档而非让开发还要自己去关注问题文档，这个步骤中还配合着DSL，利用featureid和questionid双向引用，建立机械校验映射关系，保证回灌准确度。
总的来说，自检、澄清、回灌机制带来的主要价值有：保证输出文档的粒度和覆盖度；澄清结果回灌至文档形成单一事实源；一定程度抹平不同开发对同一份prd理解偏差；最终是为了后续步骤有稳定可靠的输入文档。

追问2：14组真实样本和评估结果有什么作用？具体讲一讲review-prd和compare-prd，以及eval机制。
14 组真实样本，本质上是我给这个工具准备的一套“小型考场”。fe-design不单单是一个提示词SKILL，他需要最可能的保证真实prd投喂后输出文档质量，所以对其的后续改动不能凭增量输出结果的感觉来评判。因此我保留了14个真实prd，持续验证改动解决了一个问题，不会引起另一个场景的问题。更具体一些，这些样本的作用有几个：1.是验证泛化性，这些样本覆盖不同业务、不同复杂度；2.做版本回归，每次我有较大修改后，都拿这14组样本重新跑一遍，结合review-prd和compare-prd，看结果变好还是退化。3。沉淀数据飞轮，compare-prd会独立对比报告能力，用于沉淀数据飞轮。eval体系就是这套持续评估机制。

追问3：cli工具链深挖-这个cli工具支持哪些agent，你如何实现的适配？当用户触发cli工具命令时，发生了什么？
安装cli-> 使用/fe-design命令生成（范围过滤、放过粗拆分、自检优化，如果有待确认问题，询问澄清方式直接交互或是问题文档）-> 问题澄清 -> 回灌 -> 复盘（review-prd可选） -> 数据飞轮（compare-prd可选）。

追问4：这个和普通的提示词工程/SKILL有什么区别？
我理解 fe-design 和普通提示词工程，最大的区别是：普通提示词更像“一次性问答”，而 fe-design 已经是“可复用的生产流程”。普通提示词就是一段prompt，比较依赖个人提示词、上下文、平台，结果波动不可控，fe-design 则是把 PRD 理解这件事做成了有输入约束、流程拆分、结构化输出、澄清回灌和评估闭环的可复用能力，所以它更像工程化工作流，而不是单条 prompt。

1.  CLI 发布与多终端适配

发布机制很直接：package.json 的 bin 字段把 fe-power-prd 命令指向 bin/fe-power-prd.sh，npm install -g 后这个 shell  
 脚本就成为全局命令。

多终端适配的核心是 install 子命令，它做三件不同的事：

- Claude Code / Agents：把 skills/\*/SKILL.md 以 symlink 形式写到 ~/.claude/skills/ 和  
  ~/.agents/skills/，这两个目录是这两个平台的 skill 发现路径。symlink 的好处是仓库里改了 SKILL.md 立刻生效，不用重新安装。
- Codex：把 skills/fe-design/commands/_.md 复制到 ~/.codex/prompts/，Codex 从这个目录读取 slash command。这些文件是从  
  command-templates/_.template.md 渲染出来的，不直接手写。
- Trae：在安装时临时渲染 template 到一个 mktemp 目录，再复制到项目根的 .traecli/commands/，同时通过 trae-cli plugin
  命令注册插件。Trae 的命令是项目级的，不是全局的，所以写到工作区而不是 home 目录。

三个平台的命令内容来自同一份 command-templates/，通过 scripts/render-command-mirrors.mjs  
 渲染，保证内容一致。安装脚本还维护了一个 manifest 文件（.fe-power-prd-prompts.manifest），记录本次安装了哪些文件，卸载时按
manifest 精确清理，不误删用户自己的文件。

2. 14 组真实 PRD 样本与 eval 体系

tests/ai-seed/ 是离线 eval 数据集，14 对真实业务 PRD（prd/prd-01~14-\*.md）和对应的生成结果（vai/1~14/main/fe-design.md）。

eval 体系分两层：

第一层：确定性检查（checker.ts）。每个 PRD 对应一份 YAML  
 断言文件，声明"这个功能点必须包含哪些关键词、必须有哪些子章节、不能出现哪些词"。matcher.ts
用三层策略把生成结果的功能点和断言对齐：精确名称匹配 → 别名匹配 → 模糊关键词匹配（50% token 重叠）。对齐后 checker.ts  
 逐项做字符串级别的断言，结果确定性强、速度快、不消耗 token。

第二层：LLM 语义评判（llm-judge.ts，--llm flag 开启）。对于"语义是否正确"这类无法用关键词判断的问题，调用 Claude API  
 做语义级评判。这一层是可选的，平时跑 CI 用第一层，需要深度分析时才开。

compare-prd skill 是这套数据集的消费入口，批量模式扫描 prd/ 和 vai/ 目录、按序号配对、顺序处理 14 对并输出 reports/<timestamp>/
下的单对报告和汇总 summary.md。summary 里有高频遗漏模式和 skill 优化建议，这是反推 prompt 改进的依据，也就是"数据飞轮"的含义。

3. review-prd 和 compare-prd 的区别

两者定位完全不同，一个是实时修稿，一个是离线复盘。

review-prd 是生成链路内的内联自检。/fe-design 生成草稿后自动触发一次，AI 对照 PRD 检查四类问题（关键缺失 > 错误理解 > 额外功能 >
待确认项处理错误），能直接改的就改草稿，不确定的补进  
 questions.md，最后输出一个简短摘要。它不产出任何报告文件，只改当前这份草稿。也可以手动触发做二次复核。

compare-prd 是离线数据沉淀。它不改稿，只产出报告。单对模式给一次生成结果打分（编码安全度 0-100）；批量模式跑完 14  
 对后汇总高频问题模式，告诉你 writer 经常漏什么、question-identifier 经常误判什么，从而指导 prompt 和模板的迭代方向。

一句话区分：review-prd 是"这次交付前先自查"，compare-prd 是"跑完一批后看哪里系统性地差"。
