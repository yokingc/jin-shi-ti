STAR故事：
原生 / H5 混合交付收敛：推动小贷核心页面统一 H5 化

Situation — 背景
小贷业务早期同时存在原生页面和 H5 页面，部分链路存在多端重复开发、维护成本高、需求迭代效率低的问题。随着业务持续迭代，原生与 H5 的边界不清会导致技术债累积，也增加了问题排查和版本发布成本。

Task — 任务
我作为专项 Owner，需要推动交付形态收敛，把能 H5 化的原生页面逐步迁移到统一 H5 体系中，同时保障业务功能、性能体验、监控埋点和风险兜底能力不下降。

Action — 行动
我先梳理现有原生页面和 H5 页面清单，评估页面复杂度、业务访问量、依赖能力、改造风险和收益，确定迁移优先级。
对于高频链路，我重点确认与 App 容器能力、登录态、跳转协议、埋点、异常兜底之间的兼容性，避免简单“页面重写”带来线上风险。
迁移过程中，我推动补齐监控、埋点和异常兜底能力，并对冗余历史代码做系统性清理。为了降低改造风险，我采用分批迁移、灰度验证、核心链路回归的方式推进。

Result — 结果（带数字）
最终推动 23 个原生页面完成 H5 化，使小贷 H5 占比从 89% 提升到 99%。同时系统性下线 8.4 万+ 行冗余代码，提升了系统可维护性。项目上线后零线上问题。

标签
工程治理, H5 化, 多端统一, 技术债治理, 稳定性, 监控埋点

预测追问：

1. Q:23 个原生页面 H5 化，H5 占比从 89% 提升到 99%，这个提升如何解释？ / 为什么不是100%？ / 最难的页面是什么？
   A：最难的是风控人脸页面，经过综合评估，认为成本大于收益。核心的评估依据如下：1.风控人脸页面整个大部门都是采用原生集成的方式，交互及数据流转流程成熟，强行改造为域H5反而有差异化和心智负担 2.风控人脸页面的改动需求频率低，且涉及前端改动时往往是增加参数。3.风控人脸页面涉及多方改动，需要推动多协作部门，具有一定壁垒。换句话说，重构后，小贷业务除了风控人脸页面，其他页面最终都以H5的方式交付，因此H5占比达到了99%。

2. Q:H5 化后性能有没有下降？怎么验证？
   A：其实我司在涉及到工程治理的项目中，都有明确要求，H5性能需要达到秒开。这个统计的口径是浏览器打开页面的耗时。原生页面天生具有渲染稳定、系统能力等优势，但是小贷业务迭代频繁，原生资源协调成本高，发版远没有H5灵活。因此我们在重构为H5的过程中，把H5的性能优化也作为核心指标，缩小与原生页面的体验差异。页面性能开发阶段观测主要有两个方面：1. LCP的p75指标，开发阶段主要通过chrome devtools的performance中获取，手机调试时依赖我司基础部门提供的各阶段耗时打点，上线后还需要在监控平台观测，并且LCP内容不能是无效内容 2.视觉稳定性，关注CLS在0.1一下 3.除此之外还有白屏、交互体验、JS错误率、接口失败率等要求。

3. Q:原生能力依赖怎么处理？
   A：首先通过原生同学及业务侧共同梳理页面能力，列出原生依赖列表，然后通过技术方案评审，确认H5是否可以满足需求。系统能力依赖主要通过H5容器提供的MBbridge能力，但需要额外注意使用时的双端差异和版本兼容性，这些在bridge文档中有详细描述。这些依赖会及时同步测试同学进行回归。

4. Q:重构过程如何保证稳定性？有什么兜底方案？
   A:这类H5化重构的风险，最大的不是技术难度，而是改造后不能影响原有的业务链路。所以我们不会采用一次性替换的方式，而是按照“风险评估 -> 分阶段改造 -> 灰度验证 -> 核心链路回归 -> 监控告警 -> 可回滚兜底”的方式，确保每一步都有观察窗口和止损手段。1.重构前，先补齐监控埋点，并对页面进行分级。先改低风险页面如说明页、活动页、规则页等；而涉及核心链路的授信、还款、借款等页面，会放到第二批，且设计时保证更完整的一键回滚能力。2.灰度过程中，保留新旧链路并行。梳理的原有入口，在入口层通过配置控制用户进行原生或H5，若H5出现异常则直接切换开关。在上线稳定后，原生页面框架保留，进行兜底转发并记录来源。3.核心链路需要完整回归。 4.灰度策略：先内部白名单，小流量灰度，灰度速度根据页面风险调整。

   A：兜底方案：1.一键开关切回原生 2.页面兜底：错误页、重试、降级展示、兜底转发原生页 3.服务端配合接口兜底 4.资源兜底（CDN缓存、资源hash、记载失败监控等，联合基础部门观测）5.业务兜底，关键操作前进行校验或二次确认等操作，需要业务确认 6.发布兜底

5. Q:下线 8.4 万行代码，如何保证没有误删？
   A:1.下线前，补齐监控埋点，对页面风险进行分级，总体遵循确认无依赖，兜底方案，分批下线，灰度观察，一键回滚等来保证稳定性。

6. Q:再具体讲一讲性能优化是如何做到秒开的？做了哪些优化？
   A：1.基座层面：首先秒开很大程度上依赖我司的微前端容器能力，简单来说是让App提供一个webview预加载执行一个web项目，该web项目有运行前端项目的所有公共依赖库，这样打开业务页面时，webview就会从缓存中加载依赖库，从而提升性能。基座也融合了预加载、预渲染、预请求能力。（优点：快、统一、集中 缺点：风险高）2.业务层面：资源瘦身（下线冗余、拆分非首屏模块）缓存和资源命中（CDN缓存、文件hash）体验优化（预渲染、预加载、骨架屏等主要优化体验）异步（按需加载、异步渲染）chunk拆分

7. Q:讲一讲chunk拆分和如何验证有效？
   A:chunk切分（第三方依赖vendor、公共模块通过splitChunks.catchGroups，runtimeChunk，contenthash，minchunks，reuseExistingChunk）但是注意chunk不是越细越好，太碎容易请求过多、造成弱网并发压力、出现ChunkLoadError等，原则是首屏主包小，公共包稳定。用 webpack-bundle-analyzer：1.构建产物看main.js 2.vendor稳定 3.重复依赖 4.是否有单个chunk过大 5.source-map分析 6.首屏js加载总量 7.结合线上指标
